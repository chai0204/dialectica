## 1. プロジェクトの理念と核

### 1.1 ビジョン

このプロジェクトは、単なる知識の蓄積（Wikipedia）を超え、**情報の論理的な依存関係を可視化・構造化する「人類の知のOS」**を目指す。

### 1.2 解決する問題

現代社会の議論には以下の構造的な問題がある：

- **極端な二元論**: 複雑な問題が賛成/反対に矮小化される
- **感情的な非難**: 命題ではなく人格が攻撃される
- **不毛な蒸し返し**: 同じ議論が何度も繰り返される（前提の整理がないため）
- **前提の覆りへの鈍感**: 学術界でも基礎研究が否定されたとき、それに依存する研究群への影響が即座に把握されない
- **「当たり前」の脆弱性**: 一般に受け入れられている命題でも、実は前提に議論の余地がある場合、それが可視化されない

### 1.3 核となるアプローチ

議論を最小単位の**命題（proposition）**に分解し、それらを**グラフ構造**で繋ぐことで：

- ある命題が覆った際に、影響を受ける全ての下流命題を即座に特定できる（**覆り波及**）
- ある命題について議論する際、最も解釈が揺れている前提を優先的に表示できる（**前提脆弱性マップ**）
- 命題のもっともらしさを前提から再帰的に計算できる（**伝播計算**）
- 再利用可能な命題により情報の重複を排除できる

### 1.4 根本哲学：絶対的な真は存在しない

あらゆる命題の真偽は**確定した値ではなく、解釈の集合として存在する**。これを「量子性」と表現する。命題のもっともらしさは解釈の集合を集計して初めて算出される「観測値」であり、常に変動しうる。

## 2. 理論的基盤 — 諸学問からの知見

本プロジェクトは複数の学問領域の知見を統合している。

### 2.1 哲学・認識論

| 概念 | 提唱者 | 本プロジェクトへの適用 |
|------|--------|----------------------|
| **基礎づけ主義 vs 整合主義** | デカルト / クワイン | グラフはDAG（基礎づけ）ではなく循環を許容（整合主義）。信念の網（web of beliefs）をグラフとして実装する |
| **意味（Sinn）と指示（Bedeutung）** | フレーゲ | 表現が異なるが論理的に等価な命題を等価グループで管理し、重複を排除する |
| **推定無罪** | 法哲学 | 全ての投稿は反駁されない限り信頼される。品質判定の責任はプラットフォームではなく参加者にある |

### 2.2 言語学・意味論

| 概念 | 提唱者 | 本プロジェクトへの適用 |
|------|--------|----------------------|
| **合成性原理** | フレーゲ | 命題の「意味」はその構成要素と結合方法から一意に決まる。同じ構成→同じ命題→重複排除 |
| **フレーム意味論** | フィルモア | 各命題が暗黙に前提とする概念構造（フレーム）を明示化する。例：「売る」→ buyer, seller, goods, price |

### 2.3 論理学・形式的議論理論

| 概念 | 提唱者 | 本プロジェクトへの適用 |
|------|--------|----------------------|
| **Dungの抽象的議論フレームワーク** | Pham Ming Dung | 論証ノード + 攻撃エッジの構造。grounded semantics で「どの論証の集合が同時に成立しうるか」を計算する。反駁耐性（Layer 2）の理論的基盤 |
| **演繹 / 帰納 / アブダクション** | アリストテレス / パース | 推論パターンの型を分類するが、その妥当性の評価はプラットフォームではなく参加者が行う |

### 2.4 数学・圏論

| 概念 | 適用 |
|------|------|
| **圏（Category）** | 命題を対象、推論関係を射とする圏。射の合成が推論の連鎖 |
| **Multicategory** | 多項射（複数の前提から1つの結論）を持つ圏。ハイパーエッジに対応 |
| **n-Category** | 任意のノードがサブグラフに展開可能な再帰構造。抽象度の階層 |
| **プルバック（Pullback）** | 対立する2つの議論体系の「共有前提の発見」に対応 |
| **関手（Functor）** | 異なるドメイン間の構造的対応づけ（将来的な実装） |
| **Olog** | David Spivak（MIT）提唱の知識の圏論的構造化。型とインスタンスの区別 |

### 2.5 確率論・ベイズ推論

| 概念 | 適用 |
|------|------|
| **ベイジアンネットワーク** | 伝播もっともらしさの計算基盤。前提のもっともらしさから結論のもっともらしさを算出 |
| **最弱リンク原理** | 「AかつBが必要」な場合、min(P(A), P(B)) で集約 |
| **Noisy-OR** | 「AまたはBのいずれかで十分」な場合、1 - Π(1 - p_i) で集約 |

### 2.6 量子力学（アナロジー）

| 概念 | 適用 |
|------|------|
| **重ね合わせ状態** | 命題の真偽は確定値ではなく、解釈の集合（量子状態）として存在 |
| **観測** | 特定の文脈で解釈を集計しもっともらしさを算出する行為 |
| **量子状態トモグラフィー** | 多様な「基底」（視野）からの観測の集積が、状態の再構成精度を上げる |

### 2.7 社会心理学

| 概念 | 提唱者 | 適用 |
|------|--------|------|
| **道徳基盤理論** | ジョナサン・ハイト | 感情的反応の背後にある価値前提の分類（care/harm, fairness/cheatingなど）。ただしプラットフォームが分類するのではなく、参加者がタグとして自己申告する |

### 2.8 分散システム

| 概念 | 適用 |
|------|------|
| **ビザンチン障害耐性** | 個々のノード（参加者）が故障（偏り）を持つ前提で、全体として正しい合意に至るプロトコル設計 |
| **PageRank** | 多くの論証の前提として参照されている命題ほど構造的に重要 |

## 3. 設計原則

### 3.1 プラットフォーム = 決定論的な競技場

これは本プロジェクトの最も重要な設計原則である。

```
プラットフォームが行うこと（全て決定論的）:
  ✓ 構造の保持（グラフの保存・クエリ）
  ✓ 決定論的な計算（もっともらしさの伝播、数値集計）
  ✓ 構造的制約の強制（型チェック、循環検出、整合性保証）
  ✓ 埋め込みベクトルの計算とクラスタリング（数学的計算）

プラットフォームが行わないこと:
  ✗ 分類（「これは感情か論理か」の判定）
  ✗ 評価（「この解釈は正しい」の判定）
  ✗ フィルタリング（「この投稿は低品質だ」の判定）

LLMが行うこと（提案レベルのみ、グラフに影響しない）:
  △ 構造化の提案（「こう分解できますが、どうですか？」）
  △ 前提の提案（「この前提を仮定していませんか？」）
  △ 類似命題の提案（「これと似た命題がありますが、同じですか？」）
```

**理由**: プラットフォームが判定を行うと、プラットフォーム自体が反駁の対象になり、システムの信頼性が崩壊する。参加者は他の参加者の投稿を反駁できるが、決定論的な計算は反駁の対象にならない（同じ入力→同じ出力であり、数学的に検証可能だから）。

### 3.2 推定無罪

全ての投稿は、反駁されない限り、そのまま信頼される。品質が低い解釈も、前提のない感情的な投稿も、反駁されるまでは対等に扱われる。ただし前提を引用していない解釈は構造的健全性が低いため、もっともらしさ計算への寄与は小さい。これは「感情的だから」ではなく「構造が不足しているから」であり、決定論的な計算結果である。

### 3.3 不完全な参加の前提

各参加者（人間もAIも）はコンテクストの完全な把握を前提としない。偏った視野からの不完全な解釈が集積することで、命題の「量子状態」がより正確に形成される。不完全性は欠陥ではなく、集合知の原材料である。

### 3.4 悪意ある参加の許容

恣意的・悪意的な参加も原則として許容する。人類の総合知という点では、それも意見である。ただし、構造的な仕組みにより悪意ある投稿は自然に減衰する（排除ではなく減衰）：

- 前提なしの主張 → structural_soundness = 0.3（低い）
- 反駁されて崩壊 → refutation_resilience が低下
- 大量の同一主張 → クラスタリングで対数的に圧縮

### 3.5 全パラメータの公開

計算に使用する全てのパラメータは `system_parameters` テーブルに公開される。同じパラメータ＋同じデータ→同じ計算結果。これが「決定論的な競技場」の構造的保証である。

```
│  └───────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

### 4.2 グラフ構造の選択

**型付きハイパーグラフ**を基本構造として採用。DAG（有向非巡回グラフ）ではない。

| 特性 | 採用理由 |
|------|----------|
| **循環を許容** | 現実の知識は相互正当化（整合主義）を含む。循環は禁止ではなく検出・明示する |
| **ハイパーエッジ** | 「AかつBがCを支持する」という多項関係を1つのエッジで表現 |
| **型付きエッジ** | support, refutation, equivalence, causation, analogy 等の関係の種類を区別 |
| **再帰的ノード** | 任意の命題がサブグラフに展開可能（n-categoryの構造） |

### 4.3 差別化となる2つの核心機能

**① 覆り波及（Cascade Impact）**

ある命題のもっともらしさが大きく変動したとき、それを前提とする全ての下流命題に波及を伝播し、影響範囲を即座に可視化する。

- BFS的伝播アルゴリズム
- 減衰係数（デフォルト0.85）で遠方への影響は徐々に小さくなる
- 閾値（デフォルト0.05）以下の変動で伝播停止
- 全波及イベントを `cascade_events` / `cascade_impacts` に記録

**② 前提脆弱性マップ（Premise Fragility Map）**

ある命題について議論する際、その成立に必要な前提群を再帰的に収集し、「解釈が最も揺れている（=脆弱な）前提」を優先的に表示する。

- 脆弱性スコア = 解釈の分散 × 条件付き解釈率 × 少数解釈ペナルティ
- 深さが浅い前提ほど影響度加重が大きい
- 「広く受け入れられているが実は脆弱な前提」を炙り出す Hidden Vulnerability Detection

## 6. 計算モデル

### 6.1 2層評価モデル

解釈の信頼度（reliability）は2つの層から決定論的に算出される。

```
reliability(I) = structural_soundness(I) × 0.60
               + refutation_resilience(I) × 0.40
```

**Layer 1: 構造的健全性（Structural Soundness）— 重み 0.60**

解釈が引用する前提命題のもっともらしさから決定論的に算出。

```
structural_soundness(I) =
    前提なし → 0.3（未論証だが排除はしない）
    前提あり → weighted_avg(前提のもっともらしさ)
               × (1.0 + 0.3 × ln(前提数))
```

旧設計では推論パターンごとに基礎点を変えていた（演繹=1.0, 帰納=0.7 等）が、これはプラットフォームの価値判断であるため除去。「この推論は弱い」と思う参加者は反駁として投稿する。

**Layer 2: 反駁耐性（Refutation Resilience）— 重み 0.40**

反駁の存在と状態から決定論的に算出。

```
refutation_resilience(I) =
    反駁なし → 0.5（未検証）
    反駁あり → survival_rate × battle_tested_bonus
      survival_rate = 1 - (active反駁の構造的健全性の合計 / 正規化)
      battle_bonus  = 1.0 + min(0.3, addressed反駁数 × 0.05)
```

- 反駁されていない解釈: 中程度（まだ試されていない）
- 反駁されて耐えた解釈: 高スコア（鍛えられている）
- 反駁されて崩れた解釈: 低スコア

第3層（人間投票）は設計検討の結果、除去された。理由：「人ではなく構造を評価する」という思想との矛盾、組織票による操作リスク、設計の複雑化。

### 6.2 命題のもっともらしさ計算

クラスタベースの計算で、数の暴力を対数的に圧縮する。

```
plausibility(P) =
    Σ(cluster) [
        reliability(centroid)
        × ln(1 + member_count)
        × stance_direction(+1/-1/0)
        × cohesion
    ] / Σ(ln(1 + member_count))
```

これにより、同じ主張を100人が言っても ln(101) ≈ 4.6 の影響力しか持たない。一方、論理的に強固な少数意見は reliability スコアで大きな重みを持つ。

### 6.3 伝播もっともらしさ

前提命題のもっともらしさが変わったときの下流への伝播。

```
propagated(C) = local_weight × local(C)
              + cascade_weight × Σ_R [
                  direction(R.type)
                  × confidence(R)
                  × aggregate_premises(R)
                ]

aggregate_premises:
  - 最弱リンク: min(premises)  ← 「全て必要」な場合
  - Noisy-OR: 1-Π(1-p_i)      ← 「いずれかで十分」な場合
```

### 6.4 覆り波及アルゴリズム

```
BFS Cascade Propagation:
  1. トリガー命題を再計算
  2. delta > THRESHOLD なら波及開始
  3. 下流命題をキューに追加
  4. 各命題を再計算、delta > THRESHOLD ならさらに下流へ
  5. DAMPING_FACTOR^depth で遠方ほど減衰
  6. MAX_DEPTH で打ち切り
  7. 影響ログを記録、関係者に通知
```

### 6.5 前提脆弱性スコア

```
fragility(P) = variance_score
             × (1 + conditional_ratio)
             × (1 / ln(interpretation_count + 2))

variance_score = 4 × agree_ratio × disagree_ratio
  → agree/disagreeが拮抗するほど高い（最大0.25 × 4 = 1.0）

impact_weighted_fragility = fragility × (1 / (1 + depth × 0.3))
  → 深さが浅い前提ほど影響が大きい
```

### 6.6 Hidden Vulnerability Detection

```sql
WHERE local_score > 0.7          -- 広く受け入れられている
  AND fragility_score > 0.3      -- しかし解釈が揺れている
  AND downstream_count > 5       -- かつ多くの結論が依存
ORDER BY fragility × downstream_count DESC  -- リスクスコア
```

## 7. 不完全な参加者から総合知を創造する仕組み

### 7.1 4つの不完全性とその吸収

| 不完全性 | 発生パターン | 吸収メカニズム |
|---------|-------------|---------------|
| **前提の見落とし** | 結論だけ主張し前提を確認しない | structural_soundness = 0.3（低い）。他参加者の反駁で補完。AI参加者が前提を提案（提案レベル）。前提を追加すればスコア上昇（インセンティブ） |
| **表現の重複** | 同じことを違う言い方で大量投稿 | 埋め込みベクトルによるクラスタリング（閾値ベース・決定論的）。対数圧縮で影響力を限定 |
| **感情と論理の混在** | 「許せない」と「論理的に誤り」の未分離 | プラットフォームは感情/論理を区別しない。前提の有無で構造的に差がつく。「感情的だ」という指摘は他参加者の反駁として行う |
| **文脈の暗黙化** | 自分の前提を自覚せず明示しない | AI参加者が前提を提案（提案レベル）。他参加者が文脈の不一致を反駁。contexts テーブルでの場合分け |

### 7.2 悪意ある参加の自然な減衰

| 攻撃パターン | 減衰メカニズム |
|-------------|---------------|
| 大量アカウントで同一主張 | 埋め込みクラスタリング → ln(n) に圧縮 |
| 前提なしの感情的攻撃 | structural_soundness ≈ 0.3 → 寄与微小 |
| 形式的だが恣意的な論証 | 引用した前提のもっともらしさが低ければ Layer 1 低下。反駁で Layer 2 低下 |
| 定義のすり替え | 他参加者が定義命題レベルで反駁可能 |

### 7.3 「不完全な観測の集積」の原理

各参加者は限定された視野から偏った解釈を投稿する。個々の解釈は不完全だが、異なる偏りを持つ解釈が集積することで、命題の「量子状態」がより正確に形成される。

これは量子状態トモグラフィーと同じ原理：多様な基底（視野）からの観測の集積が、状態の再構成精度を上げる。

埋め込みクラスタリングにより、同一視野からの重複観測は圧縮され、異なる視野からの観測が自然に重視される。

## 8. LLMエージェントによるブートストラップ

### 8.1 エージェントの位置づけ

LLMエージェントは「プラットフォームの機能」ではなく「一参加者」として振る舞う。人間の参加者と完全に同じルールで計算される。AIの出力もグラフ上で反駁可能。

### 8.2 エージェント種別

| エージェント | 役割 | モデル要件 | コンテクスト |
|------------|------|-----------|-------------|
| **Seed Agent** | 既存知識（Wikipedia, 論文等）を命題として投稿 | Flash級で十分 | ~2000 tokens |
| **Dialectic Agent** | 既存命題に対して反駁を生成 | Flash級で十分（弱い反論にも価値がある） | ~1500 tokens |
| **Perspective Agent** | 異なる視野からの解釈を投稿 | ローカルLLMでも可。異なるモデルを異なる視点に割り当てると多様性が生まれる | ~1000 tokens |
| **Bridge Agent** | 異なるドメイン間の関係を発見 | やや高い推論能力が望ましい | ~3000 tokens |
| **Suggestion Agent** | 提案のみ（ai_suggestionsテーブル経由） | Flash級で十分 | 可変 |

### 8.3 コンテクストの意図的制限

各エージェントの入力は意図的に制限する。全体を把握した上で完璧な出力を出すのではなく、限られた視野から偏った出力を出す。これは人間の参加者と同じ構造であり、設計上の制約ではなく意図的な特性。

### 8.4 フェーズ計画

| フェーズ | 期間 | 内容 | 目標 |
|---------|------|------|------|
| Phase 1 | Week 1-4 | Seed Agent を集中投入 | グラフの骨格構築（~10,000命題） |
| Phase 2 | Week 2-8 | Dialectic Agent を投入 | Layer 2 データ蓄積（~20,000反駁） |
| Phase 3 | Week 3-12 | Perspective Agent を投入 | 量子状態の充実（~30,000視点解釈） |
| Phase 4 | Week 6- | Bridge Agent を投入 | ドメイン間接続 |
| Phase 5 | 継続 | 全エージェント低頻度稼働 | 自律的な成長 |

### 8.5 コスト見積もり

Gemini 2.0 Flash使用時、Phase 1-3 で約 $24。60,000ノード・100,000エッジ規模の初期データ。ローカルLLM（Llama 3 8B等）なら電気代のみ。

## 9. UX・体験デザインの方針

### 9.1 心理的安全性の確保

- 「わからない」「考慮していない」の表明を歓迎するUI
- 攻撃対象は人格ではなく命題
- `interpretation_blind_spots` で視野の限界を正直に宣言できる

### 9.2 承認欲求の転換

- 「相手を負かす」ではなく「強固なグラフを構築した」ことを評価
- ただしスコアの設計には注意が必要（些末な命題の乱造防止のため、ノードのインパクト=被参照数を評価軸に加える）

### 9.3 マイクロ貢献

- AI提案の承認/却下: 1タップ
- 暗黙前提の確認: 1タップ（✓/✗）
- 前提ギャップへの応答: 1タップ（考慮する/関係ない）
- 自由テキストの投稿: 構造化はAIが提案し参加者が承認

### 9.4 知のフロンティアの可視化

- 脆弱性マップで「まだ論理が繋がっていない領域」を可視化
- リサーチ好きの挑戦心を煽る

## 11. 残る議論の余地

### 11.1 構造・モデルに関する未解決課題

#### 11.1.1 命題の粒度制御

任意の再帰的展開は可能だが、実運用で「どの粒度で議論するか」のガイダンスが必要。異なる粒度の命題が混在した場合の計算上の扱い（抽象的な命題と具体的な命題の重みの差異）は未定義。

#### 11.1.2 圏論的「関手」の実装

異なるドメイン間の構造的対応（「経済学の議論構造と環境学の議論構造が同型」）を表現する `domain_mappings` テーブルは未実装。Bridge Agent が発見した対応をどう構造化するかの詳細設計が必要。

#### 11.1.3 文脈ごとのもっともらしさ分離

場合分けの設計として `contexts` テーブルは用意されているが、もっともらしさの計算を文脈ごとに完全に分離するか（`plausibility_cache` に `context_id` を追加）、フィルタとして処理するかは未決定。前者は計算量が文脈数に比例して増える。

#### 11.1.4 集約方法の自動選択

最弱リンク（min）と Noisy-OR の使い分けを関係の型に応じて自動選択する仕組み（`relations` テーブルに `aggregation_method` カラム追加）は設計方針のみで未実装。

#### 11.1.5 解釈のメタ解釈

「あなたの解釈は前提Xを見落としている」という指摘を、反駁として扱うか、メタ命題として扱うか。現設計では反駁（`interpretation_rebuttals`）として処理するが、重要な解釈を meta 型命題としてグラフに昇格させるパスも検討の余地がある。

### 11.2 計算・性能に関する課題

#### 11.2.1 計算コスト

`compute_interpretation_reliability_v2` は内部で複数関数を呼び、解釈数が多い命題では重くなる。解釈・反駁の変更時のみ再計算するイベント駆動設計と、`interpretation_reliability_cache` テーブルの追加が必要。

#### 11.2.2 閉包テーブル vs Recursive CTE

個人利用フェーズでは recursive CTE で十分だが、大規模化時には closure table への移行が必要。移行タイミングの判断基準は未定義。

#### 11.2.3 埋め込みモデルの選択

汎用文埋め込みモデルでは「論理的に同じだが表現が異なる」ケースの類似度が不十分な可能性がある。ドメイン特化の fine-tuning を検討すべきか、初期は汎用モデルで始めるかの判断。

### 11.3 ガバナンスに関する課題

#### 11.3.1 system_parameters の変更権限

全パラメータは公開されるが、誰がどういう手続きで変更できるのか。メタ的な議論（「この重み配分は不適切だ」）をプラットフォーム上で行い、合意に基づいて変更する仕組みは美しいが、鶏と卵の問題がある。

#### 11.3.2 等価グループの承認プロセス

「AとBは同じ命題だ」の判断は参加者が行うが、承認プロセス（何人が承認すれば確定するか等）の詳細は未設計。

#### 11.3.3 AI参加者の比率制御

AI参加者と人間参加者の比率が極端に偏んだ場合（初期は AI >> 人間）、計算結果の意味が変わる可能性。AI投稿の `user_type` は区別できるが、計算上は同一扱いとしているため、比率による補正を入れるべきかは思想的な判断が必要。

### 11.4 UX・普及に関する課題

#### 11.4.1 前提ギャップ提示のタイミング

投稿時に即座に提示すると煩わしい可能性。提示頻度の制御（3回に1回、重要度の高いギャップのみ等）のUX設計が必要。

#### 11.4.2 「些末な命題の乱造」防止

スコアを稼ぐために反駁困難だが議論的価値の低い命題（「水は100℃で沸騰する」等）が増殖するリスク。被参照数（PageRank的な重み）の導入が有効だが、詳細は未設計。

#### 11.4.3 初期ユーザーの獲得

LLMブートストラップでデータは用意できるが、人間の参加者を獲得する最初のきっかけの設計が必要。PKMツールとの連携、既存SNSでの実演、特定コミュニティ（学術界、政策議論等）への導入が候補。

### 11.5 思想的に深い未解決問題

#### 11.5.1 価値判断の扱い

`value_judgment` 型の命題に対して、もっともらしさの数値化はどこまで意味があるか。「多くの人が同意している価値判断」は「もっともらしい」と言えるか。価値の多元性を認めるならば、価値判断には数値を付けず「このような立場がある」と並列するだけが誠実かもしれない。

#### 11.5.2 「決定論的な競技場」の限界

プラットフォームは判定を行わないが、構造の設計自体が暗黙の価値判断を含む（例：「前提が多いほどスコアが高い」は「前提の明示を重視する」という価値判断）。この自己言及的な問題にどう向き合うか。

#### 11.5.3 権力構造の再生産

初期のシード提供者やパラメータ設計者が持つ影響力は、「最初に構造を定義した者の権力」を生む。プラットフォームが成長した後にこの初期構造を民主的に改変できる仕組みが必要か。
